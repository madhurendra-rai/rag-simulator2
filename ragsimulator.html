Step 2 — Chunking

Large documents are split into smaller, overlapping pieces. Smaller chunks = more precise retrieval. Overlap ensures context isn't lost at chunk boundaries — the last few words of one chunk repeat at the start of the next.

Step 3 — Vector embeddings

Each chunk is passed through an embedding model (e.g. text-embedding-3-small) which converts text into a high-dimensional numeric vector. Semantically similar texts produce vectors that are geometrically close. The visualization below shows a simplified 12-dimension representation.

Vector store

All vectors are persisted in a vector database. This index enables fast approximate nearest-neighbor search at query time.

Step 4 — Query & vector search

The user's question is embedded using the same model. The resulting query vector is compared against every stored chunk vector using cosine similarity. Higher similarity = more relevant chunk.

Step 5 — Re-ranking

Vector search returns top-k chunks by embedding similarity, but similarity isn't always the same as relevance. A cross-encoder re-ranker reads the query and each chunk together and produces a refined relevance score — often reordering the results.

Knowledge base →Chunking →Embeddings →Query + search →Re-ranking →Generation

Step 6 — Prompt construction & generation

The top re-ranked chunks are injected into a prompt alongside the original query and a system instruction. The LLM reads this combined prompt and generates a grounded answer — it cannot hallucinate facts outside the provided context.
